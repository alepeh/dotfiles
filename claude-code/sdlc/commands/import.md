# /sdlc:import — Retrofit an existing project

Brings a pre-existing repo onto the baseline **without rewriting the stack**.
Non-destructive: source code, build system, and deployment remain unchanged.
Only adds the scaffolding that makes the other `/sdlc:*` commands work.

**Scope gate:** Like `/sdlc:bootstrap`, this is for personal projects only.
If the cwd is inside a work / Paysafe repo, stop and ask the user to
confirm before proceeding.

---

## Step 1 — Detect what's already there

Check in parallel:

- `Makefile` / `Justfile`
- `architecture/` directory
- `specs/` directory (SDLC delta-merge target)
- `openspec/` directory (legacy — from older bootstraps; offer to migrate if found)
- `changes/` directory (could be ours, could be from another system)
- `.dev/` directory
- `.sdlc.yaml` (already imported?)
- `package.json` (check for workspaces)
- `wrangler.jsonc` / `wrangler.toml` (per-app)
- `.github/workflows/` (existing CI)
- `.env*` files at repo root and per-app (to seed `.env.secrets.example`)

Report a concise table of what exists vs. missing:

```
| Artifact             | Status        |
|----------------------|---------------|
| Makefile             | ✅ exists      |
| architecture/        | ❌ missing     |
| specs/               | ❌ missing     |
| .sdlc.yaml           | ❌ missing     |
| changes/             | ❌ missing     |
| wrangler.jsonc (app) | ✅ exists (2)  |
| .github/workflows/   | ✅ exists (1)  |
```

**Do not touch anything that already exists** without confirmation.

---

## Step 2 — Classify the stack

Read manifests to identify the stack:

- **TS Worker (Hono?)** — `package.json` + `wrangler.jsonc` + `src/index.ts`
- **Python Worker (FastAPI?)** — `pyproject.toml` + `wrangler.jsonc` + `src/worker.py`
- **Pages SPA** — `package.json` + `vite.config.*` + no `wrangler.jsonc`
- **Container gateway** — `wrangler.jsonc` with `containers` + `Dockerfile`
- **Non-Cloudflare stack** — e.g. pure Python CLI, Node service, Go binary

The classification shapes which capability skills are relevant:

| Stack              | Skills pulled in                                        |
|--------------------|---------------------------------------------------------|
| CF TS Worker       | cloudflare-baseline, hono-worker, local-dev             |
| CF Python Worker   | cloudflare-baseline, python-worker, local-dev           |
| CF Pages SPA       | cloudflare-baseline, local-dev                          |
| Container gateway  | cloudflare-baseline, local-dev                          |
| Non-CF             | local-dev only; flag the infra mismatch                 |

For non-CF stacks, the baseline still applies (Makefile, architecture/,
change-mgmt), but don't generate `wrangler.jsonc` or push toward D1/R2.

## Step 3 — Add missing scaffolding

For each missing piece, **ask before adding**. Add in this order:

### 3.1 — `Makefile`

Use the **local-dev** skill. If a `Justfile` exists, skip and note it;
don't duplicate. If the project already has a `Makefile` with non-standard
targets, add a new `##@ Agentic SDLC` section alongside rather than
replacing.

### 3.2 — `architecture/`

Use the **ddd-layout** skill. Seed:

- `domain-model.md` — bounded contexts extracted from the existing code
  structure. Fold top-level subdirectories of `src/` (or `apps/*/src/`)
  into candidate contexts. Flag ambiguous ones as `<TBD — confirm>`.
- `guidelines.md` — starter template. The user will customize.
- `rules.md` — empty header, ready for R-001.
- `decisions/template.md` — ADR template.
- `decisions/0001-baseline.md` — **the import ADR** (see Step 4).
- `acceptance/` — empty directory.

### 3.3 — `.sdlc.yaml`

Write at the repo root. Infer sensible defaults:

```yaml
# Generated by /sdlc:import on <YYYY-MM-DD>.
# Populate feature_groups when the project crystallizes enough to need
# formal AC tracking. Empty is fine for early-stage projects.
feature_groups: []

domain_model: architecture/domain-model.md
rules_file: architecture/rules.md
guidelines: architecture/guidelines.md
acceptance_dir: architecture/acceptance
decisions_dir: architecture/decisions
changes_dir: changes
specs_dir: specs
```

Override the default paths **only** if the project already has an
`architecture/`-like directory elsewhere (e.g. `docs/architecture/`) —
reuse, don't duplicate.

### 3.4 — `specs/`

Create `specs/` at the repo root if missing (`mkdir -p specs && touch
specs/.gitkeep`). This is the delta-merge target for feature-type changes —
no tooling required, the format is documented in the **change-protocol**
skill. If a legacy `openspec/` directory exists from an earlier import,
ask the user whether to migrate its contents into `specs/` or leave it as
reference.

### 3.5 — `changes/` (pre-existing)

If `changes/` already exists from another system (e.g. the project had its
own change-management convention), **pause and ask**. Options:

- **Merge**: if the existing `changes/` happens to match our layout (a
  `meta.yaml` per dir, archive/ subdir) — rare
- **Rename**: move existing to `changes-legacy/` and start fresh
- **Override config**: set a different `changes_dir` in `.sdlc.yaml`

Don't auto-merge or overwrite. The user needs to pick.

### 3.6 — `.env.secrets.example`

Scan existing `.env*` files at repo root and under each `apps/*/`. Extract
variable names (strip values), dedupe, write to `.env.secrets.example`
at the root. Commit this file.

Then use the **secrets-1password** skill to wire `make backup-env` /
`make restore-env` into the Makefile. Build the `ENV_FILES` list from what
you found:

```makefile
ENV_FILES := \
	.env.secrets:env-secrets \
	apps/core/.dev.vars:core-dev-vars \
	apps/email/.dev.vars:email-dev-vars
```

Don't run `make backup-env` automatically — the user should review the
scanned paths first.

### 3.7 — `.github/workflows/ci.yml`

Add the minimal CI workflow from the **cicd** skill — **only** if no CI
exists. If there's existing CI that does more than typecheck + test
(e.g. a legacy deploy workflow), leave it alone and note it in the summary.

### 3.8 — `tools/verify-ac.py`

Copy from the **change-protocol** skill's `scripts/verify-ac.py` into
`tools/verify-ac.py`. Only needed if the user plans to use AC tracking
(i.e. will populate `feature_groups` in `.sdlc.yaml` eventually). Offer
it; don't force.

## Step 4 — Capture a baseline ADR

Write `architecture/decisions/0001-baseline.md` using the ADR template.
This ADR documents **the stack as it is today** — runtime, DB,
deployment mechanism, secrets location, testing approach, CI setup.

This matters: it anchors future changes so drift from the current design
is visible. Without the baseline ADR, the project's first "real" change
has nothing to contrast against.

Example structure:

```markdown
# ADR-0001: Baseline (imported via /sdlc:import on 2026-04-17)

**Date:** 2026-04-17
**Status:** accepted
**Type:** infra

## Context

Adopting the agentic-SDLC baseline on an existing project. This ADR
captures the stack as it stood at import time.

## Decision

- **Runtime:** <e.g. Cloudflare Workers (TS, Hono) + Pages SPA>
- **Database:** <e.g. D1 per Worker, separate DBs for core and email>
- **Storage:** <e.g. R2 bucket `myproject-uploads`>
- **Deployment:** <e.g. manual via make deploy-<svc>>
- **Secrets:** <e.g. .dev.vars locally, wrangler secret put in prod,
  1Password Development vault with prefix `myproject-env`>
- **Testing:** <e.g. Vitest + pytest; Playwright lints-only in CI>
- **CI:** <e.g. typecheck + test on PR, no deploy automation>

## Consequences

- Future changes can reference this as "the baseline"
- Drift from these choices surfaces in subsequent ADRs as supersedes
- Anything not listed is an open question — flag when encountered

## Domain Impact

None — this is an infrastructure ADR.

## Rules Produced

None at import time. Rules accumulate via /sdlc:apply and /sdlc:archive.
```

---

## Step 5 — Summarize

Print:

- **Detected stack** (one line)
- **Added** — list of files/directories created, with paths
- **Skipped** — what was left alone, and why (existing file / non-CF stack / user deferred)
- **Next steps checklist:**
  1. Review `architecture/decisions/0001-baseline.md` and confirm the ADR reflects reality
  2. Review `.env.secrets.example` — did the scan miss anything?
  3. Review `ENV_FILES` in the Makefile — correct paths?
  4. Run `make backup-env` once to seed 1Password with current secrets
  5. Review `architecture/domain-model.md` — bounded contexts need real content
  6. `/sdlc:new <first-change>` to start work

Do **not** open a PR. Let the user review the diff and commit manually —
this is a scaffolding change, not a code change, and the user benefits from
seeing every file that landed.

---

## Guardrails

- Non-destructive. Never overwrite an existing file without confirmation.
- Never commit. This is a pure-add operation; the user inspects and
  commits.
- If the project is clearly not a personal project (work repo), stop at
  Step 1 and flag it.
- If `.sdlc.yaml` already exists, this is a re-import. Offer to
  re-generate only the missing pieces; don't overwrite config.
- Capability skills (cloudflare-baseline, python-worker, etc.) do most of
  the file-writing work. This command orchestrates; it doesn't re-state
  their templates.
